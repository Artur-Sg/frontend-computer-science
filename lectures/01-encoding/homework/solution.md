# Наивная схема кодирования алфавита

Наивный подход заключается в следующем:

> Каждому символу алфавита сопоставляется **уникальный фиксированный код** одинаковой длины.

Это аналогично таблице ASCII, но для расширенного набора символов.

---

# Размер алфавита

Оценим количество символов:

* кириллица: 33 буквы × 2 регистра = **66**
* цифры: **10**
* знаки препинания: ~**25**
* управляющие символы: **3**

Итого:

```
66 + 10 + 25 + 3 ≈ 104 символа
```

---

# Длина кода

Чтобы закодировать 104 символа, нужно минимум:

```
2^6 = 64  (не хватает)
2^7 = 128 (достаточно)
```

Значит, каждому символу назначается **7 бит**.

---

# Таблица кодирования

Пример (фрагмент):

| Символ | Десятичный код | Двоичный код |
| ------ | -------------: | ------------ |
| А      |              0 | 0000000      |

| ...    |            ... | ...          |
| Я      |             32 | 0100000      |

| ...    |            ... | ...          |
| я      |             65 | 1000001      |

| ...    |            ... | ...          |

(Конкретные значения можно задать произвольно — важно, что они уникальны)

---

# Кодирование текста

Каждый символ заменяется его 7-битным кодом.

## Пример

```
Привет
```

Кодируется как:

```
[П][р][и][в][е][т]
→ 7 бит на каждый символ
```

---

# Пример с цифрами

```
2025
```

```
2 → 1000010
0 → 1000010
...
```

(каждая цифра — отдельный символ по 7 бит)

---

# Свойства схемы

## Простота

* очень легко реализуется;
* не требует анализа частот.

## Однозначность

* фиксированная длина → нет неоднозначности при декодировании.

## Неэффективность

* все символы занимают одинаково 7 бит;
* не учитываются частоты (например, пробел и «о» встречаются намного чаще);
* заглавные буквы дублируют строчные.

---

# Сравнение с оптимальной схемой

| Подход           | Среднее число бит |
| ---------------- | ----------------- |
| Наивный          | ~7 бит            |
| Хаффман + режимы | ~4–5 бит          |

Потери — до **40–50% памяти**.

---

# Главные недостатки

1. Нет сжатия: частые символы не оптимизированы
2. Дублирование: верхний и нижний регистр кодируются отдельно
3. Нет структуры: цифры и знаки занимают столько же места, как буквы

---

# Итог

Наивная схема кодирования:

* проста и универсальна;
* гарантирует однозначное декодирование;
* но **существенно уступает по эффективности** схемам с переменной длиной кодов.

---

# Вывод

Наивное кодирование удобно как базовая реализация, однако для реальных задач хранения и передачи текста оно неэффективно. Более продвинутые методы (например, код Хаффмана с использованием режимов) позволяют значительно сократить размер данных.

---

# Схема более эффективного кодирования алфавита

Данные по частоте использования букв: https://ru.wikipedia.org/wiki/%D0%A7%D0%B0%D1%81%D1%82%D0%BE%D1%82%D0%BD%D0%BE%D1%81%D1%82%D1%8C

Схема строится в два уровня:

## 1. Основной уровень (префиксный код)

* кодируются **частые символы**:

  * строчные буквы;
  * пробел;
  * частая пунктуация;
  * управляющие символы;
  * служебные токены (`SHIFT`, `CAPS`, `NUM`, `PUNC`);

## 2. Режимы (модификаторы)

* используются для редких или расширенных символов:

  * заглавные буквы;
  * цифры;
  * редкая пунктуация.

# 2. Заглавные буквы

## SHIFT (одна буква)

```
SHIFT + код строчной буквы
```

Пример:

```
П = SHIFT + п
```

## CAPS (режим)

```
CAPS + текст + CAPS
```

Пример:

```
МГУ = CAPS м г у CAPS
```

# Свойства схемы

## Однозначность

* используется **префиксный код (заранее заданная таблица префиксных кодов)**;
* служебные токены явно определяют режимы.

## Экономичность

* частые символы → короткие коды;
* нет дублирования регистра;
* редкие символы не удлиняют основные коды.

## Гибкость

* можно расширять таблицу `PUNC`;
* можно добавлять новые режимы.

# Итог

Схема сочетает:

* префиксное кодирование (заранее заданная таблица);
* модификаторы (`SHIFT`, `CAPS`);
* режимы (`NUM`, `PUNC`);
* обеспечивает однозначность декодирования;
