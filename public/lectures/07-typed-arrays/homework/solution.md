## Что реализовано

1. **Часть 1 (`basic`)**
   - Формат: `count:uint32`, затем повторяющиеся блоки `len:uint32 + utf8-bytes`.
   - Методы: `encodeStrings`, `decodeStrings`, `buffer.at(index)`.
2. **Часть 2 (`pointers`)**
   - Формат: `count:uint32`, таблица `len:uint32 + pointer:uint32`, затем строка.
   - Реализовано в одном файле вместе с классом буфера-указателей.
   - Методы: `encodeStrings`, `decodeStrings`, `buffer.at(index)`.
3. **Задача со звёздочкой**
   - Добавлен `set(index, value)` в pointer-буфере.
   - `set(index, value)` реализован безопасным способом через полную пересборку буфера:
     `decodeStrings(buffer) -> изменение нужного элемента -> encodeStrings(updated)`.
   - Такой подход корректно работает со строками любой длины, включая случаи, когда новая строка длиннее старой.
   - Минус: `set` имеет сложность `O(n)`, так как пересобирается весь буфер.

## Бенчмарк (кратко)

Параметры:

- `количество измерений: 5000`
- `прогрев: 1000`
- единицы: микросекунды

| Размер | Байт в basic | Байт в pointers | pointers / basic |

## Выводы

1. `basic.at(index)` имеет линейный рост по позиции (`O(index) + декодирование`), что видно по `mid` и `-1`.
2. `pointers.at(index)` практически `O(1) + декодирование` и стабилен даже на 10k строк.
3. Компромисс ожидаемый:
   - `basic` лучше по памяти;
   - `pointers` лучше для случайного доступа.
4. Для задач с частыми `at(mid)`/`at(-1)` формат с указателями выгоднее, даже при увеличении объёма буфера.
5. Текущий `set` не оптимален по скорости: в pointer-формате его можно ускорить через append новой строки в конец буфера и обновление `ptr` в таблице.
